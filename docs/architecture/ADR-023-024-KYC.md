# ADR-023 — Estrategia de almacenamiento de documentos KYC

## Metadata
- **Feature:** FEAT-013 | **Fecha:** 2026-03-24 | **Estado:** Aceptado
- **Supersede:** ninguno
- **Normativa:** RGPD Art.9 · AML Directive EU 2018/843

## Contexto

El módulo KYC requiere almacenar documentos de identidad (DNI, NIE, Pasaporte) de hasta 10MB.
Estos documentos constituyen datos especiales según RGPD Art.9 (datos biométricos e identificativos)
y deben tratarse con medidas técnicas reforzadas: cifrado en reposo, control de acceso estricto,
trazabilidad de acceso y retención mínima de 5 años.

El proyecto BankPortal opera en fase de MVP con infraestructura on-premise en Banco Meridian.
No existe contrato cloud activo para almacenamiento de objetos.

## Decisión

**Almacenamiento en filesystem local cifrado con AES-256-GCM, path configurable via `KYC_STORAGE_PATH`.**

Cada fichero se cifra en escritura con una clave AES-256 derivada de `KYC_ENCRYPTION_KEY` (env var),
se calcula y persiste su SHA-256 para verificación de integridad en cada lectura, y se almacena
con nombre basado en UUID (no en el nombre original del fichero del usuario).

## Opciones consideradas

| Opción | Pros | Contras |
|---|---|---|
| **Filesystem local cifrado (elegida)** | Sin dependencia cloud · Implementación en sprint · Control total cifrado · Cumple RGPD | No escalable horizontalmente · Backup manual necesario |
| S3 / Blob Storage cloud | Escalable · Backup automático · Redundancia | Requiere contrato cloud + ADR nuevo + DPA con proveedor · Fuera de scope Sprint 15 |
| Base de datos (BYTEA PostgreSQL) | Simple · Sin filesystem | Degrada rendimiento PostgreSQL · Ficheros hasta 10MB impracticables en BD relacional |

## Consecuencias

- **Positivas:** implementable en Sprint 15 sin infraestructura nueva · cumple RGPD Art.9 · SHA-256 garantiza integridad
- **Trade-offs:** almacenamiento no distribuido — si hay múltiples instancias del backend, `KYC_STORAGE_PATH` debe ser un volumen compartido (NFS o similar)
- **Riesgos:** pérdida de datos si el volumen falla sin backup → Banco Meridian debe configurar backup del volumen antes de PROD
- **Impacto en servicios existentes:** ninguno — módulo nuevo aislado
- **Migración futura:** la interfaz `DocumentStorageService` permite sustituir la implementación por S3Adapter sin tocar los use cases (Hexagonal Architecture)

## Implementación técnica

```java
// Escritura — AES-256-GCM
SecretKey key = new SecretKeySpec(hexToBytes(kycEncryptionKey), "AES");
Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
cipher.init(Cipher.ENCRYPT_MODE, key, new GCMParameterSpec(128, generateIv()));
byte[] encrypted = cipher.doFinal(fileBytes);
// Prefija IV (12 bytes) + ciphertext → escribe a disco

// Hash SHA-256
String sha256 = Hex.encodeHexString(
    MessageDigest.getInstance("SHA-256").digest(fileBytes));
```

# ADR-024 — Validación automática asíncrona de documentos KYC

## Metadata
- **Feature:** FEAT-013 | **Fecha:** 2026-03-24 | **Estado:** Aceptado

## Contexto

La validación automática de documentos KYC (verificación de caducidad, tipo reconocido, integridad)
puede tardar hasta 5 segundos según RNF-D13-06. Ejecutarla síncronamente dentro de la request HTTP
de subida degradaría la experiencia del usuario (timeout percibido) y acoplaría dos responsabilidades
en el mismo flujo de control.

## Decisión

**Validación asíncrona mediante Spring `@Async` + `ApplicationEventPublisher`.**

Al completar la subida, `UploadDocumentUseCase` publica un `KycDocumentSubmittedEvent`.
`ValidateDocumentUseCase` escucha el evento con `@EventListener` + `@Async` y ejecuta
la validación en un thread pool separado. La respuesta HTTP 201 se devuelve inmediatamente
tras persistir el documento — sin esperar la validación.

## Opciones consideradas

| Opción | Pros | Contras |
|---|---|---|
| **Spring @Async + Events (elegida)** | Desacoplado · Sin infraestructura nueva · En proceso · Simple | No persistente — si el proceso muere entre subida y validación, se pierde el evento |
| Kafka / RabbitMQ | Persistente · Escalable · Retry automático | Infraestructura nueva · Fuera de scope Sprint 15 |
| Síncrono en request | Simple | Degrada UX · Timeout HTTP · Acoplamiento de responsabilidades |

## Consecuencias

- **Positivas:** UX fluida (HTTP 201 inmediato) · sin infraestructura nueva · implementable en sprint
- **Trade-offs:** eventos en memoria — si el servidor se reinicia entre subida y validación, el documento queda en SUBMITTED sin validar (operador lo revisará manualmente)
- **Mitigación:** job `@Scheduled` diario que revalida documentos en SUBMITTED > 1h sin validar
- **Impacto en servicios existentes:** ninguno
